# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Использован датасет `S12-hw-dataset.csv`, прогнозировалась числовая целевая переменная `target`.
- Выполнено корректное разбиение данных по времени: `train (70%), validation (15%), test (15%)` в хронологическом порядке.
- Сравнивались четыре подхода: наивный baseline B1 (`naive-last`) (прогноз по последнему значению), скользящее среднее B2 (`moving-average`), линейная модель Ridge на лаговых и календарных признаках B3 (`ridge-lag-features`), и рекуррентная нейросеть GRU R1 (`gru-forecast`).
- Лучшим подходом по валидационной метрике MAE оказалась модель `GRU R1 (gru-forecast)`, которая показала наименьшую ошибку на валидации (best_val_mae = 5.773).

Сложность прогноза заключается в необходимости улавливать как краткосрочные (суточные) колебания, так и долгосрочные тренды.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: данные отсортированы по дате, затем выполнено последовательное разбиение: первые 70% строк — train, следующие 15% — validation, последние 15% — test.
- Какие доли или интервалы использовались: train — 70% (первые 3024 наблюдения), validation — 15% (следующие 648 наблюдений), test — 15% (последние 648 наблюдений).
- Почему `random split` здесь некорректен: при случайном перемешивании нарушается временная структура ряда, что приводит к утечке будущей информации в обучающую выборку и создаёт иллюзию высокой точности модели, которая не сможет работать на реальных временных рядах.

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались:  lag_1, lag_7, lag_14 (лаги 1, 7, 14 часов назад).
- Какие rolling-признаки использовались: rolling_mean_7 (скользящее среднее за 7 часов), rolling_std_7 (скользящее стандартное отклонение за 7 часов).
- Какие календарные признаки использовались: день недели, месяц, а также их циклические кодировки (sin/cos) для учёта суточной и недельной периодичности.
- Как обрабатывались пропуски после построения лагов/rolling: начальные строки, где признаки не могли быть вычислены, удалялись.
- Использовалось ли масштабирование и где именно оно обучалось: для модели `B3` использовался `StandardScaler`, обученный только на train-части; для `GRU` масштабирование `StandardScaler` применялось к входным последовательностям также с fit на train.

## 5. Модели и эксперименты (B1, B2, B3, R1)

Опишите коротко и сопоставимо:

- B1 (`naive-last`):  прогноз на следующий шаг равен последнему известному значению (lag_1). Метрики считались на validation.
- B2 (`moving-average`): прогноз равен среднему значению за последние 7 часов (rolling_mean_7). Метрики считались на validation.
- B3 (`ridge-lag-features`): линейная модель с L2-регуляризацией, обученная на лаговых, rolling- и календарных признаках. Признаки нормализовались, масштабирование обучалось на train. Метрики считались на validation.
- R1 (`gru-forecast`): однослойная GRU с hidden size = 48, прогнозирующая следующее значение на основе последовательности длины window_size = 28. Использовался StandardScaler для нормализации входных последовательностей, оптимизатор Adam с learning rate = 0.001, batch size = 64. Метрики считались на validation и test.

Дополнительно:

- Основная метрика выбора лучшей модели: `MAE (best_val_mae)`
- Дополнительные метрики: `RMSE, MAPE`
- `window_size` для `GRU`: 28
- Batch size: 64
- Максимальное число эпох: 12
- Optimizer: Adam
- Learning rate: 0.001
- Как сохранялась лучшая модель: state_dict лучшей модели в `./artifacts/best_gru.pt`

## 7. Анализ

Наивный baseline (B1) показал достаточно высокую точность (val_MAE = 6.443), что говорит о том, что часовой ряд обладает высокой автокорреляцией — ближайшее прошлое (предыдущий час) действительно является хорошим предиктором следующего часа. Однако на резких поворотах тренда и в моменты смены суточных паттернов этот подход даёт сдвиг прогноза.

Скользящее среднее (B2) оказалось худшим среди всех подходов (val_MAE = 12.724), поскольку оно сглаживает все колебания и даёт систематическое отставание от реальных значений. Для часовых данных этот метод неприменим.

Ridge-модель на признаках (B3) существенно улучшила результат (val_MAE = 6.441) за счёт включения лагов 7 и 14 (часов), а также rolling-статистик, отражающих локальную волатильность. Календарные признаки (день недели, месяц) помогли модели адаптироваться к повторяющимся суточным и недельным паттернам, характерным для часовых данных.

GRU (R1) показала наилучший результат (val_MAE = 5.773), что объясняется способностью рекуррентных сетей автоматически извлекать долгосрочные зависимости из последовательности без ручного конструирования признаков. Окно в 28 часов позволило модели улавливать суточные циклы (24 часа) с небольшим запасом, что особенно важно для часовых рядов с выраженной суточной сезонностью.

На тестовой выборке ошибка GRU ожидаемо выросла (test_MAE = 7.018 против val_MAE = 5.773), что характерно при переходе на новые, невидимые участки ряда.

Утечки данных могли бы возникнуть при использовании random split (информация из будущего попадала бы в обучение) или при построении признаков с заглядыванием вперёд (например, если бы rolling-статистики рассчитывались с учетом будущих данных). В работе все преобразования выполнялись с соблюдением временного порядка.

Ошибки модели на тесте показывают, что GRU хуже справляется с прогнозированием резких скачков и пиковых значений, что является типичной проблемой для прогнозных моделей на волатильных часовых рядах. Однако она стабильно улавливает общий тренд и основные периодические компоненты.

## 8. Итоговый вывод

Для данной задачи (прогнозирование часового ряда) наилучшим подходом оказалась GRU, которая смогла уловить как суточную и недельную сезонность, так и долгосрочный тренд, превзойдя ручные признаки. В качестве базового решения для такого ряда можно рекомендовать Ridge-модель на лаговых признаках — она проще в реализации, быстрее обучается и даёт результат лишь незначительно хуже нейросети (разница в MAE около 9%).

Простых baseline-моделей (наивный прогноз, скользящее среднее) недостаточно для часовых данных со сложной сезонной структурой. Переход к GRU или другим рекуррентным сетям оправдан, когда ручной инжиниринг признаков не позволяет уловить сложные временные зависимости (например, взаимодействие суточных и недельных циклов), а вычислительные ресурсы позволяют обучать нейросеть.

Главный вывод по валидации временных рядов: нарушение хронологического порядка при разбиении приводит к нереалистично оптимистичной оценке качества моделей, поэтому для любых задач прогнозирования необходимо использовать temporal split и строго следить за отсутствием утечек данных из будущего. Для часовых данных это особенно критично, так как модели могут "подглядывать" в будущие часы и создавать иллюзию высокой точности.
